# Dimak Prospector — v2 plan

v1 finds **who to contact** (live companies from Google Places). v2 adds a
**market-intelligence layer**: *which Gulf market to prioritize for steel doors*,
based on real import statistics. It sits alongside the prospect list; it does not
change any v1 behavior.

## The data input: steel-door imports by Gulf country

**What:** annual (and, where available, monthly) **import** figures for
**HS `730830`** — *"Doors, windows and their frames and thresholds for doors, of
iron or steel"* — for each Gulf country, plus year-over-year growth so we can see
where demand is rising. HS `730830` is the steel-door code (fire doors share it),
so this is scoped **to steel doors only**, as requested.

### Where we pull the data

**UN Comtrade API** — the free, official source. This is the same trade data that
sits behind ITC's TradeMap, but TradeMap itself is login-gated with no open API
and terms that don't allow scraping, so we go straight to the source.

- Register once for a **free API key** at
  https://comtradedeveloper.un.org/ (select the free `comtrade - v1` product).
  Free tier: **500 calls/day, 100k records/call** — we need ~6 calls/month.
- Store the key as a server-side env var **`COMTRADE_API_KEY`** (never exposed to
  the browser; same pattern as the Places key).

**Request shape** (one call per reporter country, or batched):

```
GET https://comtradeapi.un.org/data/v1/get/C/A/HS
    ?reporterCode=<country>   # Gulf reporters, see below
    &period=<year>            # e.g. 2024 (or a comma list of recent years)
    &partnerCode=0            # 0 = World (all trading partners)
    &flowCode=M               # M = imports
    &cmdCode=730830           # steel doors
    &subscription-key=<COMTRADE_API_KEY>
```

Gulf reporter codes (UN M49 numeric):

| Country       | Code |
|---------------|------|
| UAE           | 784  |
| Saudi Arabia  | 682  |
| Qatar         | 634  |
| Kuwait        | 414  |
| Oman          | 512  |
| Bahrain       | 48   |

Key fields from the response we keep per country/year: `primaryValue` (import
value, USD), `netWgt`/`qty` (quantity), `period`, `reporterCode`. From the last
few years we derive **latest import value** and **YoY growth %**.

> Data caveats to surface in the UI: trade stats lag (the latest complete year is
> usually 1–2 years back), some small states report irregularly (we fall back to
> "mirror" data — what the *world* reports exporting *to* them — when a country's
> own report is missing), and values are customs-declared, not company-level.

## Data model (new table, separate from `prospects`)

```
Table steel_door_imports:
  country       text        -- 'UAE', 'Saudi Arabia', ...
  reporter_code integer     -- UN M49, e.g. 784
  period        integer     -- year, e.g. 2024
  hs_code       text        -- '730830'
  import_value  real        -- USD (primaryValue)
  quantity      real        -- net weight or qty, nullable
  is_mirror     boolean     -- true if derived from partner (mirror) data
  source        text        -- 'UN Comtrade'
  fetched_at    timestamptz default now()
  primary key (reporter_code, period, hs_code)
```

This keeps market data cleanly separate from company prospects (different grain:
country-year vs. company). It also generalizes: adding wooden doors later is just
another `hs_code`.

## API surface

- `GET /api/market` — returns the Gulf countries ranked by latest steel-door
  import value, each with YoY growth and the year, read from Postgres (cheap,
  no external call).
- `POST /api/market/refresh` — admin action: pulls the last ~3 years from UN
  Comtrade for the six reporters and upserts `steel_door_imports`. Behind the
  password gate.

## Refresh strategy

Trade data changes annually (occasionally quarterly), **not** per user action, so
we never call Comtrade on a page load. Instead:

- A **Vercel Cron** job (e.g. monthly) hits `/api/market/refresh`, or
- The user clicks a "Refresh market data" button when they want it.

Either way the numbers are cached in Postgres and served instantly to the UI.

## UI

A compact **"Market — steel door imports"** strip on the console: the six Gulf
countries as cards ordered by import value, each showing the latest value, the
year, and a ▲/▼ growth badge — so Firat can see at a glance which markets to
work first. Optionally, annotate the city chips (e.g. a small "top market"
marker) to tie prioritization back to the search.

## Suggested build phases (v2)

1. Register the Comtrade key; add `COMTRADE_API_KEY` to `.env.example` and Vercel.
   Add the `steel_door_imports` Drizzle table + migration (apply via Neon SQL
   editor, same as v1).
2. `lib/comtrade.ts` (fetch + normalize) and `POST /api/market/refresh`; verify
   one manual refresh populates the table on the deployed app.
3. `GET /api/market` + the market strip UI.
4. Vercel Cron for the monthly auto-refresh.
